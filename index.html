0. Overview & Fundamentals

What is the Internet

  • Network, internet and Internet:

    • 网络(network)：由节点(node)和链路组成(link)；节点可以是端系统或者交换设备；链路可以是有线或无线；

    • 互联网(internet)：网络的网络，也就是范围大些的网络；

    • 因特网(Internet)：最大的互联网；

  1. 边缘与核心的“物理构成”描述 (关注 Internet 是由什么硬件和软件组成的)；

    • 网络边缘： 连接在互联网上的所有设备(电脑、手机、服务器)，即端系统 (end system)；

    • 网络核心： 互联端的链路(光纤、铜线等)和分组交换机(路由器、链路层交换机)；

    • 协议：如TCP、IP、HTTP；

  2. 面向应用的“服务提供”描述 (将 Internet 视为一个为分布式应用提供通信服务的基础设施)；

    • 支撑平台： 为 Web、视频流、社交媒体等各种分布式应用提供运行环境；

    • 通信接口 (API)： 为端系统上的程序提供一套规则，让它们能够像使用“邮政系统”一样，将数据发送给另一个程序；

    • 服务级别： 提供可靠(如 TCP)或尽力而为(如 UDP)的传输选择；

Communication Concepts

  • 带宽：R ，链路每秒传输的bit数(每秒推入链路的bit数)，单位有 bps, Mbps (可能会叙述为 (发送)速率R)；

  • 传输时延：dtrans ，将数据推进链路的耗时， dtrans=数据长度/带宽R；

  • 传播速度：s ，数据在链路中传播的速度；

  • 传播时延：dprop ，数据从链路一端到另一端的耗时， dprop=链路长度/传播速度s;

  • 带宽-延迟积： R×dprop ，链路中同时容纳的bit数，即链路的比特容量；

  • 比特宽度： Lbit=链路长度/(R×dprop)=链路长度/(R×链路长度传播速度)=传播速度s/带宽R ；

    ⟹ 带宽R=链路中每秒传输的bit数=传播速度s/比特宽度Lbit ；

  • 信道利用率 = 数据发送时间 / 总周期时间，即 U=dtrans/(dtrans+2dprop)=dtrans/(dtrans+RTT)；

  • 吞吐率(bps)，单位时间内通过某个网络 (或信道/接口) 的数据量 = 传输总数据量(bit) / 传输时间(s)；

End of 0.

1. Application Layer

1.1. Principles of Network Applications

Network Application Architectures

  1. 客户 - 服务器体系结构：服务器总是打开的，并且具有固定的、周知的IP地址；客户相互之间不直接通信；数据中心：托管大量主机，用于创建强大的虚拟服务器；

  2. P2P 体系结构：应用程序在间断连接的主机对之间直接通信(称为对等方)，不依赖专门的服务器；具有自扩展性；高度非集中式结构的问题：安全性、性能、可靠性；

Transport Services Available to Applications

  • 传输层向应用层提供的服务：进程到进程的通信；

    • TCP：可靠的，字节流服务 (不维护报文边界)；

    • UDP：不可靠、无连接的消息边界明确的数据报服务；

End of 1.1.

 

1.2. Application-Layer Protocols

  • 应用通过交换报文进行通信，而应用层协议规定了：

    • 交换的报文类型 (如请求报文和响应报文)；

    • 各种报文类型的语法 (报文中的各个字段及这些字段是如何描述的)；

    • 字段的语义 (字段中信息的含义)；

    • 确定一个进程何时以及如何发送报文，对报文进行响应的规则；

  • 应用层协议只是网络应用的一部分；

The Web and HTTP

  • Web：

    • 全称 World Wide Web，即万维网；

    • Web三大支柱：HTML(超文本标记语言)，HTTP(超文本传输协议)，URL(统一资源定位符)；

      • Web页面(也叫文档)由对象组成；

      • 对象是一个文件(比如HTML文件，JEPG图像，CSS样式表文件等)，可通过一个URL寻址；

      • 一般来讲一个Web页面包括一个HTML基本文件和若干个引用对象；

  • URL由两部分组成：存放对象的服务器主机名 和 对象的路径名;

  • HTTP：

    • HyperText Tranfer Protocol，HTPP超文本传输协议；

    • HTTP由两个程序实现，客户程序与服务器程序，运行在不同端系统中，通过交换HTTP报文进行会话；

      Web浏览器实现了HTTP客户端，所以术语浏览器和客户基本等价；

      Web服务器实现了HTTP服务端，存储Web对象，每个对象由URL寻址；

    • HTTP协议采用的传输层协议：TCP；

    • HTPP是无状态协议，不存储关于客户的任何信息；

DNS

  • 主机的两种标识方式：主机名标识 与 IP地址标识；

  • DNS (Domain Name System 域名系统) 提供的服务：主机名到IP地址转换的目录服务；

    • DNS的两个部分：一个分层的DNS服务器实现的分布式数据库；一个使得主机能够查询分布式数据库的应用层协议；

  • 域名命名逻辑： 子域名 . 二级域名 . 顶级域名

    • 顶级域TLD可分为通用顶级域gTLD(.com .net .org) 、国别域ccTLD(.cn .us) 和行业/品牌顶级域(.edu .gov)；

  • 二级域：自定义的真正注册部分；

  • 子域名：www官网、text测试、blog博客...；

  • DNS域名解析流程：

    1. 本机查询：浏览器查缓存 → 查操作系统DNS缓存 → 查本地Hosts文件；都没有 → 下一步；

    2. 本地DNS服务器查询：自己机器随机端口 → DNS服务器53端口；然后本地DNS查自己缓存，若没有 → 下一步；

    3. 本地DNS服务器发起递归查询

      根域名服务器 → 返回顶级域(如.com)服务器地址；

      顶级域服务器 → 返回example.com的权威DNS服务器地址；

      权威DNS服务器存有域名到IP地址的映射，把最终结果返回给本地DNS服务器；

    客户端只问一次本地 DNS 服务器，所有查询由本地 DNS 服务器完成，最后直接把 IP 交给客户端；

  • DNS采用的传输层协议是UDP；

End of 1.2.

2. Transport Layer

  • 传输层：提供运行在不同主机上的进程间(逻辑)通信；

  • 传输层协议工作在端系统上；

    • 发送端：传输层接收来自应用层的报文，并封装为传输层的报文段(可能还会分块)，然后交给网络层；

    • 接收端：网络层提取出报文段并向上交付给传输层，传输层处理报文段再交付给目的进程(通过socket)；

2.1. Multiplexing and Demultiplexing

  • 传输层的多路复用与多路分解：将网络层提供的主机到主机的交付服务延伸到为运行在主机上的应用程序提供进程到进程的交付服务；

  • 传输层实际上不是直接将数据交付给进程，而是交给中间的套接字socket；

socket

  • 一个 socket 实际上就是内核里的一个数据结构 + 一组API接口；

    一块内核内存区域：记录了IP、端口、状态、缓冲区(发送队列与接收队列)，记录的进程PID等；

  • 两种socket：

    • UDP套接字仅通过 (目的IP，目的端口号) 标识，内核不区分发送者；

      但应用进程通过 recvfrom() 系统调用，可以获取报文的源 IP 和源端口，从而区分来自不同发送者的报文；

    • TCP已连接socket用四元组：(目的IP，目的端口号， 源IP，源端口号) 标识；一个TCP socket 只能一对一通信；

    注：TCP和UDP报文段都只有源端口号和目的端口号字段，因为源IP和目的IP在IP数据报中；

  • 所以多路分解 (将运输层报文段中的数据交付到正确的套接字) 的原理就是：报文段带有协议、源端口号，目的端口号等字段，提取该字段并交付给对应socket即可(一般通过Hash)；

  • 多路复用：在源主机从不同套接字中收集数据块，并为每个数据块封装上首部信息 (以后用于分解) 从而生成报文段，然后将报文段传递到网络层；这个过程倒是不怎么关心端口号，因为已经封装好了；

End of 2.1.

 

2.2. Connectionless Transport: UDP

  • 仅在IP协议的基础上增加了运输层最基础的 多路分解 / 复用 和 少量差错检测；

  • 无连接：没有三次握手，发送方直接交付数据，尽力而为传输，不保证交付、不排序、无重传机制；

  • 应用场景：适合对实时性要求高、能容忍少量丢包、不希望传输延迟的场景；

UDP Segment Format

  • 首部：固定长度，8 字节(4 个字段，每个字段 2 字节 / 16 比特)；

  • 数据字段：承载应用层数据，长度可变；

UDP Checksum

  • 发送方计算：将 UDP 报文段(首部 + 数据)按 16 比特分组；对所有 16 比特字做二进制加法，溢出位回卷(即加到结果的最低位)；对和做反码运算，结果填入检验和字段；

  • 接收方校验：将收到的完整报文(含检验和)按 16 比特分组求和；结果全1 → 无差错，校验通过、含0 → 有差错，校验失败；

    • 发现差错后直接丢弃报文，或交付应用层告警，不重传；

End of 2.2.

 

2.3. Connection-Oriented Transport: TCP

  • 点对点连接，一次发送操作中不能从一个发送端将数据传送给多个接收端；

  • 可靠的，按顺序的字节流：数据交付给应用层(写进内核缓冲区时)不保留报文界限(黏包与短读问题)；

  • 管道化(流水线)：TCP流量控制和拥塞控制设置窗口大小；

  • 发送和接收缓存 (为协议设计而存在，并非仅仅平衡速率)；

  • 全双工连接：同一连接中，数据可以双向流动，MSS：最大报文段大小；

  • 面向连接：数据交换前，通过握手(交换控制报文)初始化发送方接收方的控制变量；但连接是逻辑的，仅两端维护连接状态，中间网络设备并不感知连接的存在；

  • 有流量控制：发送方不会淹没接收方；

  • 有拥塞控制；

TCP Segment Format

  • TCP报文段由首部和数据组成；

  1. 首部：

    • 各16bit的源端口号和目的端口号字段 → 实现多路复用与分解(见上)；

    • 各32bit的序号和确认号字段 → 实现可靠数据传输；

    • 4bit的首部长度字段：单位为32bit(4byte) → 首部最长 4×1111(2)=60 字节；

      选项字段为空时 → 首部长为典型的20字节；

    • 12bit的各种标志字段(有4bit保留未用)：

      • ACK：指示确认字段有效；

      • RST、SYN、FIN → 连接管理 (连接的建立和拆除)；

      • CWR、ECE：拥塞通告；

    • 16bit的接受窗口字段 → 实现流量控制；

    • 16bit的因特网校验和字段 → 检查bit级的差错；

  2. 数据：

    • 来自应用层的报文，可能被拆分，导致接收方短读；

    • 说明TCP是面向字节流的协议，不维护报文界限 (需要应用自己维护)；

Maximum Segment Size

  • 最大报文长度 (Maximum Segment Size, MSS) → TCP报文段中数据的长度；

  • MSS = MTU(链路层最大传输单元) - (IP首部 + TCP首部)；

    以太网MTU一般是1500字节，如果IP首部和TCP都是最小的20字节，那么MSS就是1460字节；

    注意以太网MTU不包括帧头和帧尾，专指上层载荷最大长度；

  • MSS初始值来源于三次握手期间：

    • 连接双方在SYN报文(d)中互相通告期望的MSS值；

    • 每一方发送数据时，会选择对端通告的MSS值与自身路径限制两者中的较小值；

  • TCP连接过程中MSS一般不变，但是：

    • 路径MTU发现：某路由器发现需要分片但是IP头部设置了DF标志，则丢弃该分组并向源主机发送ICMP报文进行通告；

    所以MSS反映的是链路承载能力，不会因为两端(比如接收窗口满了)而变化；

TCP Flow Control

  • 发送方发送速率 > 接收方读取速率 → 接受缓存溢出；

  • 假设接收方丢弃失序的报文段；则对于接收方，需保证：

    LastByteRevd−LastByteRead≤RevBuffer

    RevBuffer：接受缓存；

    LastByteRead：接收方应从缓存中读取的数据流的最后一个字节；

    LastByteRevd：从网络中到达且已被接收方缓存的数据流最后一个字节；

    设接收窗口 rwnd=RcvBuffer−[LastByteRevd−LastByteRead]；

    开始时接收方设定 rwnd=RevBuffer，并在后续通信中将报文段中接收窗口字段值设定为 rwnd；

  • 发送方需控制：

    LastByteSent−LastByteAcked≤rwnd

    就可保证接收方缓存不溢出；

    LastByteSent−LastByteAcked 就是发送方已发送到连接中但是未被确认的数据量(也就是发送窗口)；

    如果加上拥塞控制，则 LastByteSent−LastByteAcked≤min{rwnd,cwnd}；

  • issue：当接收方 rwnd=0 并已通知发送方，若此后接收方没有数据要发给发送方，那么发送方将一直认为接收方缓冲区满；

    • TCP规范要求：当接收方接收窗口为0时，发送方继续发送只有一个字节数据的报文段，接收方对这些报文段进行确认；这些确认报文段可以通告发送方后续 rwnd 的变化；

End of 2.3.

 

2.4. TCP Connection Management

Establishing a Connection: The Three-Way Handshake

  1. 客户端 → 服务端

    • 发送第一次握手包，无应用层数据，SYN比特被置1 (称为 SYN报文段)；

    • 客户随机选择自己的初始序号 (记为client_isn)，置于SYN报文段的序号字段中；

    • 选项字段中带有MSS；

    作用：客户端探测服务端是否可以进行通信，端口是否打开；

  2. 服务端 → 客户端

    • 服务器收到SYN报文段，分配TCP缓存与变量；

    • 发送第二次握手包，无应用层数据，SYN比特被置1 (称为 SYNACK报文段)

    • 确认号字段值为client_isn+1；

    • 服务器随机选择自己的初始序号 (记为server_isn)，置于第二次握手包的序号字段中；

    作用：告知客户端可以发，并且服务端可以收；

  3. 客户端 → 服务端

    • 客户端收到SYNACK报文段，分配TCP缓存和变量；

    • 发送第三次握手包，可以包含应用层数据，SYN比特被置0；

    • 确认号字段值为server_isn+1；

    作用：客户端收到第二次握手包，得知客户端可以发，并且服务端可以收；于是发送第三个握手包，告知服务端可以发，客户端可以收；

→ 自此建立了全双工的四个操作通道，分别为 客户端收发、服务端收发；

SYN握手过程中除了确定双方初始序号，还会通告MSS，是否开启时间戳等；

Terminating a Connection: The Four-Way Wave

  • 由于 TCP 是全双工的 (即双方都可以同时收发数据)，关闭连接时需要双向确认；假设 Client 发起关闭请求：

  1. 第一次挥手：Client → Server (FIN)

    客户端发送一个 FIN (Finish) 报文段，进入 FIN_WAIT_1 状态；Client不再发送数据，但仍可接受数据；

  2. 第二次挥手：Server → Client (ACK)

    服务器收到 FIN 后，回传一个 ACK并进入 CLOSE_WAIT 状态，客户端收到ACK后进入 FIN_WAIT_2 状态；

    • 此时处于“半关闭”，客户端不再发服务器也不再收，但服务器如果还有没发完的数据，仍可以继续发送；

  3. 第三次挥手：Server → Client (FIN)

    服务器发送完所有数据后(如果有)，也会向客户端发送一个 FIN 报文；Server也不再发数据，并进入 LAST_ACK 状态；

  4. 第四次挥手：Client → Server (ACK)

    客户端收到服务器的 FIN 后，发送最后一个 ACK ，进入 TIME_WAIT 状态；服务器收到该 ACK 后立即进入 CLOSED 状态；客户端在等待 2MSL (最大报文生存时间) 后，也进入 CLOSED 状态；

    如果第四次挥手的 ACK 丢失，服务器会超时重传第三次的 FIN；如果客户端立即关闭，就无法处理这个重传的 FIN，导致服务器无法正常进入 CLOSED 状态；

为什么是4次？

  • 关闭是单向的： 收到 FIN 只意味着对方不再发数据，不代表自己也没数据要发了；

  • 中间的等待： 第二次和第三次挥手之间，服务器可能还有缓冲区数据需要处理；

End of 2.4.

 

2.5. TCP Reliable Data Transfer

TCP Timeout Boundary

  1. 移动平均值：

    EstimatedRTT=(1−α)⋅EstimatedRTT+α⋅SampleRTT (α 常取0.125)；

    每采集一个SampleRTT，前面的 RTT 对平均值的贡献权重就会再乘一个 1−α，指数下降；

  2. 偏差估计：

    DevRTT=(1−β)⋅DevRTT+β⋅|SampleRTT−EstimatedRTT| (β 常取0.25)；

    采样值离平均值的偏差程度 的移动平均值；

  3. 超时界限：

    TimeoutInterval=EstimatedRTT+K⋅DevRTT(K=4)

    直接用移动平均值作为 timeout 是不合适的，需加上4倍 DevRTT 进行修正；

构建可靠数据传输协议；

TCP Reliable Data Transfer

  • TCP采用：

    • 单一的重传定时器：只针对最早的未确认分组；

    • 累计确认：接收方回传ACK555代表正确接收554及之前的所有字节，希望发送方从555开始发送；

若忽略流量控制与拥塞控制，且上层数据长度小于MSS，单方向传输；

  • 此刻TCP需要响应三个事件：

    1. 上层新数据到达：TCP将其封装为报文段再交给IP(即进入发送窗口)

      → 若此刻定时器休眠(发送窗口为空)，则按照 TimeoutInterval 启动；

      → 若定时器正在运行(发送窗口不为空)，则不更新定时器；

    2. 超时：只重传最早的在途未确认分组，并加倍定时器；

    3. ACK到达：

      → ACK小于窗口后沿：丢弃；

      → 大于：窗口直接滑动到ACK序号，滑动后窗口不为空就再次重启定时器(针对处于窗口后沿的分组，也就是窗口中最早的在途未确认分组)；

  • 注1：超时 间隔加倍机制

    • 单个分组若始终未被确认 (自身 ACK 或后续分组累计确认)，超时重传定时器会指数加倍；

    • 新数据入窗不影响定时器，超时仍翻倍，直至收到ACK；

    • 收到ACK后窗口滑动，剩余未确认分组按正常RTT重启定时器，不再加倍；

    也就是说，旧分组长期无ACK会导致窗口阻塞；而累计确认可一并确认旧分组，恢复正常计时；

  • 注2：快速重传机制

    • 接收方回传ACK时机：

      1. 接受窗口为空时，若有报文段到达，不立即回传ACK，而是等待500ms看有没有新的到达；若有则一起确认，没有只确认已经到的；

      2. 失序报文段到达，接收窗口前沿和到达报文段之间产生“空腔”，立即发送“冗余ACK”，表明期望的报文段没有按序到达；

      3. 到达的报文段填补了空腔，接收窗口滑动，立即发送ACK表明期望的下一个报文段；

    • 其中情况2会是产生冗余ACK的主要原因，比如期望2，但是3，4，5都到了，此时发送方会收到3个冗余的ACK2，此刻哪怕定时器没到时，发送方也会重传序号2的报文段(快速重传)；

End of 2.5.

 

2.6. TCP Congestion Control

拥塞控制原理；

  • 虽然现代TCP也会采用网络辅助的拥塞控制，但是经典的TCP还是采用端到端的拥塞控制；

    因为标准中没有规定IP层要向传输层提供网络拥塞反馈；

  • TCP/IP架构主要将复杂性放在端系统上；

三个问题：

  1. 发送方如何调整发送流量的速率；

  2. 发送方如何感知网络的拥塞程度；

  3. 发送方感到拥塞时，什么算法能改变发送速率；

Adjust the sending rate

  • 设置拥塞窗口 cwnd；

    则 LastByteSent−LastByteAcked≤min{rwnd,cwnd}；

    接下来的讨论中忽略接收窗口rwnd的限制；

  • 则发送方最多能一次发送 cwnd 个字节的数据，并在一个 RTT 后收到确认报文；

    • 则发送速率为 cwnd/RTT 字节每秒；由此，调节 cwnd 的值就能调整发送速率；

Congestion awareness

  1. 丢包(超时 or 3个冗余ACK) → 网络拥塞 → 发送方减少拥塞窗口长度；

  2. 发送方收到ACK → 网络正常 → 发送方增加拥塞窗口长度；

    • 所以拥塞窗口增加速率取决于ACK到达速率 → TCP被称为是自计时的；

TCP congestion control algorithm

设发送方每次发送 cwnd 个字节的数据，并在一个 RTT 后收到确认报文；

  • 三个状态：慢启动，拥塞避免，快速恢复；

  1. 慢启动：

    • TCP连接开始时，cwnd被设定为一个MSS；慢启动阈值ssthresh被设定为一个很大的值；

    • ACK确认了几个报文段(累计确认)，cwnd就增加几个MSS；所以慢启动阶段每过一个RTT，速率就翻倍；

      开始cwnd长为一个MSS，只能发送一个报文段并在一个RTT后被确认，然后cwnd变成两个MSS；然后就可以发两个报文段ACK一次性确认两个，然后cwnd变成4个MSS；....；所以慢启动并不慢，反而是指数级增加；

    但是这个指数增长不是无限制的：

    A. 发生超时导致丢包：将ssthresh设为当前cwnd的一半，cwnd重置为1个MSS并重启慢启动；

    B. 收到3个冗余ACK，TCP执行快速重传，并进入快速恢复；ssthred设置为cwnd的一半，cwnd减半；

    C. cwnd增加到ssthresh，那么结束慢启动，并进入拥塞避免；

  2. 拥塞避免：

    • 没出现超时或者丢包，cwnd顺利增加到ssthresh，结束慢启动并进入拥塞避免；

    • 此时每个ACK仅让cwnd增加1个MSS(无论确认了多少个报文段)；

    当然这样的线性增加也不是无限制的：

    A. 发生超时导致丢包：将ssthresh设为当前cwnd的一半，cwnd重置为1个MSS并重启慢启动；

    B. 收到3个冗余ACK，TCP执行快速重传，并进入快速恢复；ssthred设置为cwnd的一半，cwnd减半；

  3. 快速恢复：

    • 收到3个冗余ACK执行快速重传并进入快速恢复，ssthred设置为cwnd的一半，cwnd也减半；

    • cwnd先加上3个MSS，用于补偿刚才的3个冗余ACK；之后每收到一个冗余ACK，cwnd就+1个MSS；

    • 直到收到新的ACK，退出快速恢复；降低cwnd至ssthred，进入拥塞避免，开始每个RTT增加一个MSS的线性增长；

注：TCP早期版本不区分超时丢包和冗余ACK指示，都是直接ssthresh减半、cwnd置1、慢启动；

End of 2.6.

 

 

Part Two: Network Layer

  • 网络层：提供主机到主机通信 (best effort, 尽力而为的服务)；

    为实现该服务，网络层具备的两个核心功能：转发(单台路由器) & 路由(整个网络)；

  • 网络层因此被分成两个互相作用的平面：数据平面与控制平面；

Data Plane & Control Plane Overview

  • 网络层所有工作都可以浓缩为两个核心动作(转发 & 路由)，分别属于两个不同平面：

    术语转发与交换是等价的，都指“把分组从输入端口送到输出端口”；

    分组交换机：通用术语，指按首部字段转发分组的设备，分两类：

    • 链路层交换机(2 层设备)：看链路层帧的字段转发；

    • 路由器(3 层设备)：看网络层IP数据报的字段转发；

    本章的“端口”一词不同于传输层协议中的“端口”；

    • 路由表，存在于控制平面，全局网络范围；

    • 转发表：存在于数据平面，是路由表的部分镜像，本地范围；

Implementation Paradigm: Coupling vs. Decoupling

  1. 传统方式：Per-router Control 每路由器控制

    数据平面与控制平面耦合在同一台路由器中，垂直集成；

    • 路由器既执行本地转发，也通过本地的控制平面组件根据全局网络拓扑计算生成与分享路由表；

    • 数据平面和控制平面的组件是相同的，都是路由器，每台路由器都具备转发和路由的功能；

    弊端：难以更新与升级，也不灵活(无法实现不同类别流量的分流)；

  2. SDN 软件定义网络

    解耦数据平面与控制平面；

    • (逻辑上)集中的SDN控制器统一执行路由功能：计算和下发路由表；

    • 路由器仅保留简单代理，负责接收控制器指令；

Evolution of Forwarding Mechanisms

  1. 传统转发

    • 仅依据目的 IPv4 / IPv6 地址；

    • 采用最长前缀匹配；

    • 核心表项：转发表；

    • 核心动作：转发；

  2. 泛化转发

    • 多维字段匹配：如MAC、IP、Port、VLAN ID等；

    • 采用完全匹配或通配符匹配；

    • 核心表项：流表；

    • 动作：转发、block、泛洪等等；

    • 可自定义链路负载，实现流量工程；

注：流表通过定义不同的 匹配-动作，实现了设备身份的统一：

  • 如果流表只匹配“目的 IP”并执行“转发”，它就等同于路由器的转发表；

  • 如果流表只匹配“目的 MAC”并执行“转发”，它就等同于交换机的 MAC 地址表；

  • 如果流表匹配“五元组”并执行“丢弃”，它就等同于防火墙的 ACL (访问控制列表)；

  • 如果流表匹配“VLAN 标签”并执行“修改”，它就实现了 VLAN 转换或 NAT；

End

 

3. The Network Layer — Data Plane

3.1. Architecture of a Router

  • 四大核心组件

    • 数据平面：输入端口，交换结构，输出端口；

    • 控制平面：路由选择处理器；

    输入/输出端口是按照功能的逻辑划分，实际上路由器端口既是输入端口又是输出端口；

Input Port

;

  • 输入端口有三层动作：

    1. 物理层：光 / 电信号 → 比特流；

    2. 链路层：解帧 → 取出 IP 数据报；

    3. 网络层：查找转发表以确定分组的输出端口并将其送入转发结构；

  • 转发表来源：路由选择处理器 (使用路由选择协议与路由器进行交互) / 远程SDN控制器；

  • 本地分布式转发：每个输入端口的线路卡都拥有转发表的影子副本；

  • 查找规则：最长前缀匹配；既能聚合路由，又能保障转发的精确；

  • 概念上查找是简单的，但是：

    1. 为了极致性能，需要采取特殊搜索策略(线性搜索太慢)，需要使用特殊硬件元件(TCAM)；

    2. 为了功能完善，在查找前需要进行物理层和链路层处理(三层动作)，需要检查分组的版本号、检验和以及寿命字段(还需要重写后两个字段)，需要更新用于网络管理的计数器；

  • 如果一个分组在进入交换结构时交换结构正忙，则该分组会被阻塞，并进行排队与调度；

Switching

;

  • 将分组实际地从输入端口转发到输出端口；(术语“转发”等价于“交换”)；

1) 经内存 memory 交换

  • 早期做法：

    • 结构：所有端口共享一块内存和一个CPU；

    • 转发流程：输入端口 → 内存(写) → 输出端口(读)；

    • 性能瓶颈：

      • 每个分组需要两次I/O操作 (输入端口到内存，内存到输出端口)；

      • 若内存每秒最多处理 B 个分组读/写，则总吞吐量 ≤ B/2；

      • 共享系统总线，内存操作串行化，无法并行转发多个分组；

  • 现代路由器也是采取经内存交换的方式：

    • 将查表与写入输出端口内存下放到输入线路卡硬件来完成；

      输出端口拥有逻辑上独立的内存区域，输入线路卡可直接写入目标输出端口内存；

    • 系统特征：

      • 类似分区共享内存的多处理器系统；

      • 多线路卡可并行转发，显著提高吞吐量；

2) 经总线 bus 交换

  • 转发流程：

    • 输入端口查转发表，确定输出端口，给分组加上内部标签，并送上总线；

    • 所有输出端口并行监听，标签匹配则接受分组，再去掉内部标签，发送到外部链路；

  • 交换带宽受限于总线速率：

    • 总线带宽有限，每个时钟周期只能传输固定数量的bit；

    • 总线访问是互斥的：同一时刻只能一个分组占用总线，其它分组会被阻塞，因此分组在总线上是串行通过的；(这里的串行指的是时间，总线本身一个时钟周期内能并行32/64/128bit..)；

3) 经互联网络 crossbar 交换

  • 纵横式交换的核心结构：

    • N 条垂直总线 + N 条水平总线组成，连接 N 个输入端口和 N 个输出端口；

    • 每条垂直总线和水平总线的交叉点有开关，由交换控制器控制是否闭合；

  • 转发流程：

    • 分组到输入端口 A，查表要去输出端口 Y，于是控制器闭合 A 与 Y 的交叉点开关；

    • 端口 A 在自己的垂直总线上发送分组，只有 Y 水平总线能收到 (只有这个交叉点通)；

    • 如果同时，另一个分组从 B → X：控制器闭合 B 和 X 的交叉点，两条路径完全独立、并行传输；

  • 特性：并行 + 非阻塞

    • 只要交叉点不冲突，多个输入输出流就可以并行；只要目的输出端口不被占用，新的分组就不会被阻塞；

    • 唯一冲突：两个分组的输出端口相同时；

Output Port

;

  • 选择并取出排队的分组，然后进行链路层和物理层处理；

  • 当分组到达速率 > 输出端口发送速率时，分组会被阻塞，并进行排队与调度；

  • 两个问题：排队 & 调度；

    • 一般不考虑输入队列的调度，输入队列最主要问题是队头阻塞；

Problem1. Queuing

  • 输入端口和输出端口都可能形成排队；

    当队列太长导致缓存耗尽时，分组会被丢弃，也就是丢包；

  1. 输入排队：

    • 假设输入端口采取先到先服务 (FCFS) 方式，当输入队列的队首分组被阻塞 (对应输出端口正忙)，那么队列后面所有分组都将被阻塞，即使它们对应的输出端口空闲；这就是队首阻塞 (Head of the Line (HOL) blocking)；

  2. 输出排队

    • 当输出端口的缓存耗尽时：

      1. 丢弃新到达的分组 (称为弃尾策略)；

      2. 删除先前的某些分组以腾出空间；

      3. 或者是在缓存即将耗尽时就开始丢弃分组 / 标记分组(向发送方发送拥塞信号)；

      • 关于上述分组的丢弃与标记已经提出了多种策略，统称为主动队列管理 (AQM)；

        最广泛的AQM算法是 随机早期检测，队列还未溢出但超过某个阈值时就开始随机丢少量包，让TCP降速；

  • 当输出队列形成时，分组调度器将选择某个分组进行传输；

  1. 缓存大小问题

    • 少数TCP流的缓存大小经验公式：缓存数量 = RTT × 链路容量； B=RTT×C；

    • 大量TCP流： B=RTT×C/N；(N 是TCP流数量)；可见大量流会使得所需缓存明显减少；

    • 缓存不是越大越好，对实时性敏感的应用，宁愿丢几个包，也不让它排队增加链路传播时延；

    • 缓存膨胀 (Bufferbloat) 是指由于网络设备中超大容量缓冲区被低速链路上的冗余流量填满，导致数据包因长时间排队而产生的持续、高量级的端到端时延；

Problem2. Output Port Scheduling

  • 先进先出：FIFO，也称先来先服务 FCFS；

  • 优先权排队：

    • 优先传高优先权队列中的分组 (同一优先权类的分组采用FIFO)；

    • 非抢占式优先权排队：高优先权分组的到达不会打断正在传输的低优先权分组；

  • 循环排队：分组仍然被分类，但是不存在优先权，调度器轮流循环地传输各个队列的分组；

  • 加权公平排队(WFQ)：

    • 最广泛使用的调度规则；

    • 分组仍然被分类，但是每个类会被分配一个权重 wi；

      • 即使所有类都有分组排队，类 i 也至少能得到 wi/∑wj 的带宽；

End of 3.1.

 

3.2. IPv4: The Internet Protocol

IPv4 Datagram Format

  • 版本：4bit，用于确定IP协议的版本(IPv4/IPv6)；

  • 首部长度：由于IP首部的“选项”字段是可选的，所以需要首部长度字段；

  • 服务类型 (TOS)：区分不同类型的IP数据报 (实时 vs 非实时等)；

  • 数据报长度：用于确定整个IP数据报的长度；

    注意TCP头部仅有首部长度字段；

  • 标识，标志与偏移 → 分片机制有关 (IPv6不允许分片)；

    标识：是否属于同一个分组；标志：是否允许分片，后面是否还有分片；

  • 寿命：数据报每经过一跳，寿命TTL-1，TTL为0时该数据报被抛弃(并且向源主机回传ICMP控制报文；

  • 上层协议：指明数据部分按哪个传输层协议解析，值6为TCP，值17为UDP等等；

  • 首部检验和：用于检验IP首部是否出现bit差错，注意仅针对IP首部，而传输层协议的检验和是针对于整个传输层报文段；

  • 后面两个就是源与目的的IP地址；

  • (IPv6没有选项字段)；

如果没有选项字段，则IP首部为20字节，如果携带TCP报文段(也没有选项)，则报文段首部也为20字节，共40字节；

IPv4 Fragmentation

  • IPv4头部中有一个Flags(标志) 字段；共 3 个bit：

    1. Bit 0： 保留位(Reserved)，为 0；

    2. Bit 1：DF (Don't Fragment) 位；

    3. Bit 2：MF (More Fragments) 位(表示后面还有分片)；

  • 当一个 IP 数据包到达某台路由器时，路由器需要将该包转发到下一跳链路；

    • 此时，路由器会对比数据包的总长度与输出接口的 MTU；

    • 如果 数据包长度 > 接口MTU，路由器会读取该包报头中的 DF 位：

  1. DF = 0 (允许分片)

  • 路由器将该大包切割成多个满足 MTU 限制的小分片，并为每个分片复制报头、修改 Fragment Offset(片偏移)和 MF 位，然后转发出去；

  1. DF = 1 (禁止分片)

  • 路由器执行：

    1. 丢弃该包；

    2. 向该包的源 IP 地址发送一个 ICMP 差错报文(Type 3, Code 4 需要分片但设置了DF位)；

    3. 在该 ICMP 报文中，路由器会附带当前输出接口的 MTU 值；

但是现代网络抛弃了路由器分片机制(IPv4的DF位默认为1)； IPv6 中直接禁止路由器分片，只允许端到端分片；

现代网络依赖一种机制：路径 MTU 发现(PMTUD, Path MTU Discovery)；

  • 如果路由器发现数据包长度 > 接口MTU，就丢包(因为DF=1)并返回上述的 ICMP Type 3, Code 4 报文；

  • 源收到该ICMP报文，悉知其附带的Next-Hop MTU，其网络栈会修改该连接的 MSS；

  • 此后 TCP 就会将按照该MSS切割上层报文；

IPv4 Addressing

  • IPv4地址：32位；采用点分十进制记法；

  • IPv4地址标识的是物理接口而不是设备；

  • IPv4地址分配：无类别域间路由选择 CIDR

    • 用物理链路相连 / 无线接入点互联 的接口构成一个子网；

    • 子网地址a.b.c.d/x 的前 x bit称为该地址的网络部分或前缀，/x 称为子网掩码；

  • IPv4寻址：

    • 路由聚合：例如子网 192.168.2.0/24 与 192.168.3.0/24 可聚合为 192.168.0.0/22；减少转发表的长度；

    • 最长前缀匹配：有子网越过聚合网关直接与上层网关相连时，或是子网同时接入多个网关时；

  • 路由器转发动作：将目的地址与掩码按位与，结果与子网地址相同则转发；

    • 0.0.0.0/0 表示默认路由，因为任何地址与0按位与结果都是0，必然与默认路由匹配；

Dynamic Host Configuration Protocol (DHCP)

  1. DHCP服务器发现：

    客户端采用广播目的地址255.255.255.255和源地址0.0.0.0，UDP报文段，目的端口67；称为DHCP发现报文；

    封装成IP数据报再交给链路层，链路层将该帧广播到所有相连节点；

  2. DHCP服务器提供：

    DHCP服务器收到DHCP发现报文时，用DHCP提供报文进行响应；

    将事务ID，分配的IP地址，地址租用期等等广播到255.255.255.255；

  3. DHCP请求：

    客户可能收到多个DHCP提供，选择一个，发送DHCP请求报文进行响应(带有事务ID，MAC地址等以区分)；

    服务器发现不是响应自己就回收分配的IP地址；

  4. DHCP ACK：

    服务器用DHCP ACK确认DHCP请求报文；

Network Address Translation (NAT)

  • NAT：在数据包通过路由器时，修改 IP 报文头部的地址信息；

    • 出站 (Outbound): 将内部私有 IP替换为唯一的公网 IP；

    • 入站 (Inbound): 根据记录好的映射表，将公网发来的回包转发回对应的私有 IP；

  • NAT采用端口号区分不同设备与不同进程；

  • 中间盒设备：工作在网络层，但是不同于路由器执行传统的数据报转发，而是执行NAT、流量流的负载均衡、防火墙等等功能；

End of 3.2.

 

3.3. IPv6

IPv6 Datagram Format

  • 首部为定长的40字节；

  • 流量类型：对应IPv4头部中的服务类型字段；

  • 流标签：发送方随机生成的20位数值，用于标识一个特定的“流” (Flow)；

    • 流通常指从特定源地址到特定目的地址、具有相同传输层端口号的一串连续报文 (即五元组：源 IP、目的 IP、源端口、目的端口、传输层协议)；

  • 有效载荷长度：标识了数据字段的大小；

  • 下一个首部：指明数据部分按哪个传输层协议解析，与IPv4的上层协议字段值相同；

  • 跳限制；

  • IPv6的地址长度拓展为128bit；且除了单播与多播地址，还引入了任播地址(任意交付给一个)；

IPv6禁止路由器的分片行为，只允许端到端分片；

From IPv4 to IPv6

  • 采用隧道技术 (Tunneling Technology)；

  • 隧道技术中，原始数据包(包括原始协议头)会被视为净荷(Payload)，再次包裹上一个新的隧道网络报头；

    • IPv6-in-IPv4 隧道中，外层IPv4头部的上层协议字段值为41；隧道出口处路由器可根据该值获悉这是一个隧道包 (中间路由器不感知)；

  • 运载协议头会带来开销 → 隧道设备会拦截SYN报文段，缩小其中的MSS (见2.4.)；

End of 3.3.

 

 

4. The Network Layer — Control Plane

4.1. Routing Algorithms

  • 要求正确性、简单性、健壮性(Robustness)；

  • 最终得到每个节点的汇集树 (Sink Tree)；

  • 链路代价的确定及其更新机制；

Classification Method

  • 分类方式1：集中式 or 分布式；

    1. 集中式路由选择算法：算法拥有全局状态信息，因此也称作 Link-State 算法；

    2. 分散式路由选择算法：每个节点只维护与其直接相连链路的开销，通过迭代、分布式的计算方式得到最短路径；

      典型的分散式路由选择算法：Distance-Vector 算法；

  • 分类方式2：静态 or 动态；

    1. 静态路由选择算法：路由随时间变化非常缓慢；

    2. 动态路由选择算法：随着网络流量负载或拓扑发生变化而改变路由选择路径；

  • 分类方式3：负载敏感 or 负载迟钝；

    1. 负载敏感算法：链路开销会动态地变化以反映出底层链路的当前拥塞水平；早期算法采用，问题也较多；

    2. 负载迟钝算法：某条链路的开销不明确地反映其当前的拥塞水平；当今路由算法(OSPF / BGP)采用；

注：路由表三个表项：目的节点、总开销、下一跳；

The Link-State (LS) Routing Algorithm

  1. 节点获取全局网络拓扑

    • 采用链路状态广播 (link state broadcast) 算法，每个节点都洪泛(flooding)其链路状态分组 (link-state packets)；

      而DV算法中节点仅向物理邻居发送路由表；

  2. 所有节点得到一致的全局拓扑后 → 两种LS算法：Dijkstra算法 or Prim算法；

  • 当边代价代表该链路拥塞程度时，可能出现的问题：路由震荡；

The Distance-Vector (DV) Routing Algorithm

  • 特点：迭代、异步、分布式、自我终止；

  • 节点向所有物理邻居发送本地路由表 (不包含路径信息)，同时根据邻居路由表更新本地路由表；

  • 核心更新逻辑：Bellman-Ford 方程

    dx(y)=minv{c(x,v)+dv(y)}

    x 是源节点，y 是目的节点， v 是 x 的任一邻居节点，c(x,v) 为链路代价；

  • 无穷大问题：邻居短路径可能包含节点本身，由此引发路由选择环路，缓解方法为 毒性逆转 等；

  • 此后 本地路由表变化时 / 周期性地 向邻居发送路由表，最终网络中无路由表交换 (前提是网络拓扑不变)，所有表稳定，算法自我终止；

End of 4.1.

 

4.2. Intra-AS Routing in the Internet: OSPF

  • 自治系统 (Autonomous System, AS)：指拥有统一路由策略的、由巨大网络和路由器组成的集合；

    • 唯一的 ASN 自治系统编号；

    • CIDR IP 路由前缀；

  • OSPF (Open Shortest Path First，开放式最短路径优先) 是最主流的内部网关协议 (IGP) 之一；

OSPF

  • 链路状态路由协议；

    • 还解决了传统距离矢量协议 (如RIP) 收敛慢、易环路、无法适应大规模网络等问题；

  • 工作原理：三大核心组件

    1. 邻居表：记录已建立邻居关系的路由器；定期发送Hello报文来维系；

    2. 链路状态数据库 (LSDB)：所有路由器会生成链路状态宣告 (LSA，描述自己的接口状态和开销)，并泛洪到整个区域；最终同一区域内的所有路由器会拥有相同的 LSDB (完整的网络拓扑图)；

    3. 路由表：路由器以自己为根运行最短路径优先算法 (SPF，基于Dijkstra)，计算到达每个网段的最短路径，并交给本地的全局路由表；

    为什么需要全局路由表？因为路由信息不止OSPF产生，可能还会来自于网络管理员静态配置等等；

  • OSPF 封装在 IP 协议之上，协议号为89 (所以有LSAck报文确保OSPF传输的可靠性，因为不基于TCP)；

  • 区域 (Area)：用于减少SPF计算开销和LSDB数据量；

    • 骨干区域 (Area 0)：必须存在且唯一；与所有非骨干区域相连；

    • 1类和2类LSA只在区域内泛洪；跨区域时，区域边界路由器会将其转化为3类LSA；

  • 特点：

    • 每台 Router 的ID唯一；

    • 所有OSPF报文必须认证 (安全性)；

End of 4.2.

 

4.3. Routing Among the ISPs: BGP

  • 边界网关协议 (BGP)

    • 路径矢量协议 (非距离矢量)：将整个AS视作黑盒，路由信息为该路由穿过的AS路径序列 (解决环路问题)；

  • BGP实际是将分组路由到CIDR化的前缀(子网)；

  • BGP的两个功能：

    1. 确保每个AS都知道所有子网；

    2. 能得到达到每个子网的最佳路由选择；

BGP Route Advertisement Mechanism

  • 路由器角色：

    • 网关路由器 (Gateway Router)： 位于 AS 边缘，与相邻 AS 的路由器物理相连；

    • 内部路由器 (Internal Router)： 仅在本 AS 内部进行交换；

    现代网络中，内部路由器一般不需要运行BGP，而是通过 MPLS (多协议标签交换) 来承载BGP流量；

    或者通过0.0.0.0/0来指向默认网关；

  • BGP 会话：

    • BGP路由器通过 TCP port 179 交换路由信息 (TCP 保证了路由通告的可靠性)：

    • eBGP (External BGP)： 跨越 AS 边界的会话；

    • iBGP (Internal BGP)： 同一 AS 内部的会话；

  • 路由信息通告逻辑：

    1. 当AS获知一个新子网时，其网关会通过eBGP向相邻AS发送通告；

    2. 网关收到 eBGP 报文后，会将该路径信息通过 iBGP 告知 AS 内部所有BGP路由器；

      补充：iBGP 并非简单的泛洪，通常要求 AS 内所有 BGP 路由器建立全连接 (Full-mesh)，以确保路由信息的一致性；

    3. 其他BGP路由器得知路径后，会继续向其相连的其他 AS 网关通过 eBGP 扩散该路由；

      补充：这种扩散受到策略 (Policy) 的控制 (e.g. AS 可能不想为某个邻居充当流量中转站)；

  • 路径矢量 (Path Vector) 协议特性：

    1. 路径属性 (Path Attributes)：

      BGP 通告的不仅是目的地，还附带了属性：

      • AS-PATH： 路由经过的所有 AS 编号列表；

      • NEXT-HOP： 到达目的地应发往的下一个特定网关的 IP；

    2. 与 DV 算法的区别：

      • 防环路 (Loop Prevention)： BGP 路由器如果发现 AS-PATH 中包含自己的 AS 号，会直接丢弃该路由；

      • 策略选路： BGP 选路除了代价考量，更多是基于商业协议或管理策略 (e.g. 优先走免费的路径而非付费的路径)；

Determine the best route

  • 前面说过，BGP通告包括目的地前缀，AS-PATH，NEXT-HOP；

  1. 路由器还会被指派一个本地偏好，最高本地偏好的路由将被筛选；

  2. 筛选出的路由中，最短AS-PATH的路由将被进一步筛选；

  3. 余下的路由中，采用热土豆路由进行选择 (让分组尽量快得离开本AS，而不关心AS间路由代价)；

  4. 如果还有多条路由，则使用BGP标识符进行选择；

“选择”具体是指每接收到一个路由选择就与原有路由选择进行比对，更优则更新，否则丢弃；

End of 4.3.

 

4.4. The SDN Control Plane

SDN Controller

  • SDN控制平面分为 SDN控制器(网络操作系统) 和 SDN网络控制应用程序；

    • 控制器通过北向API为控制应用提供网络视图，通过南向API来操纵交换机流表；

  • SDN控制器又分为：

    • 上层接口：采用北向API沟通网络控制应用和网络范围状态管理器；

    • 中间为网络范围状态管理层：维护网络拓扑，为网络控制应用提供必要信息，并接受控制应用的策略转为具体流表(通过北向API)；

    • 下层为通信层：下发流表给网络交换机，回传网络交换机产生的网络状态信息(通过南向API)；

OpenFlow

  • 运行在TCP之上，默认port 6653；

  • 实现了网络设备控制平面与数据平面的分离：

    • SDN 控制器利用 OpenFlow 协议，向底层的南向网络设备直接下发、修改或删除流表；

    • 传统的分布式路由协议 (如 OSPF、BGP) 在单个设备上被停用；网络设备成为纯白盒的转发实体，完全听从 OpenFlow 传达的控制器最高指令；

    • 允许控制器实时收集底层硬件的流量统计数据和拓扑状态变化，从而让控制平面能够基于全网全局视角计算出最优的数据路由与控制策略；

End of 4.4.

 

4.5. The Internet Control Message Protocol (ICMP)

  • IP 协议缺乏差错报告和控制机制；当网络出现故障或需要诊断时，使用 ICMP 来进行通信；

  • ICMP 报文主要用于以下两类任务：

    • 差错报告：当路由器或主机在处理 IP 数据报发现错误时 (如TTL耗尽)，会向源主机发送 ICMP 报文；

    • 网络探测：探测网络状态，如测试目标是否在线或测量往返时间；

ICMP Message Format

  • ICMP 报文封装在 IP 数据报 的数据部分中(上层协议字段为1)；包含以下字段：

    • 类型 (Type)：报文的大类；

    • 编码 (Code)：在类型基础上进一步细化原因；

    • 校验和 (Checksum)：用于检测报文在传输过程中是否损坏；

    • 引起该ICMP报文首次生成 的 IP数据报的首部 和 前8个字节 (以便发送方能确定引发该差错的数据报)；

  • 常见的 ICMP Type & Code：

    • Type 0 Code 0：Ping的回送应答；

    • Type 3 Code 4：需要分片但设置了 DF 标志；

    • Type 11 Code 0：TTL耗尽；

  • ICMP 是网络层协议；

End of 4.5.

 

4.6. Network Management and SNMP, NETCONF/YANG

 

Part Three: Data Link Layer & Wireless Networks

5. Data Link Layer and LANs

  • 链路层：在相同链路或局域网之间更局域化地移动分组；

5.1. Error-Detection and -Correction Techniques

  • 情形设定：

    • 发送方欲发送数据 D，包括网络层数据报及链路层帧首部；

    • 发送方使用差错检测和纠正比特 (Error-Detection and-Correction, EDC)来保护 D；

Parity Checks

  • 单奇偶校验：为 d 位数据添加一个额外bit位(称为奇偶校验位 parity bit)，使得这 d+1 中比特1个数为偶数；接收方发现1个数是奇数说明出错了(也可以反着来)；

    • 多bit翻转时检测效果不佳；

  • 二维奇偶校验：将数据分成多行多列，每一行每一列都予以一个奇偶校验比特；

    • 单bit翻转时不仅能检测到，还能定位到具体的行和列 (可纠错)；

    • 同行或者同列多bit翻转，虽能检测出差错，却不能纠正；

    • 比特翻转不在同行同列且恰好对偶，就检测不出来了；

Checksum

传输层协议常用 (TCP / UDP)；

  • 将数据 D (共 d 位bit) 视为一系列 k 比特的整数序列，并将这些整数相加；

  • 检测逻辑：(以因特网检验和为例)

    将数据的字节作为16比特的整数进行求和，取其反码作为检验和存放在报文首部；

    接收方对包括检验和在内的所有接收数据求和并取反，如果结果全是 1，则认为没有差错；若出现 0，则说明存在错误；

  • 易通过软件高效实现，但是保护效果较弱；

Cyclic Redundancy Check (CRC)

循环冗余校验，原理暂不知....

Example: 设待保护数据为 00101001，约定的生成多项式 X3+X2+1；

  1. 展开多项式得到二进制CRC除数：多项式为 R=3 阶，展开得到除数为 1101；

  2. 补零：在数据后补 R 个零：00101001000；

  3. 模2除法计算余数：

    .

    • 待保护数据的前导零不用管；

    • 下放几位数是看结果前面有几个零；

    • 计算终点：结果有效位数少于除数时；

  4. 得到 R 位结果：001 (补零或减零)；

  5. 将结果拼接到待保护数据后，就是最终发送的数据：00101001001；

关于第3步中计算终点的提醒：

假设CRC除数有4位，某次结果为0001，应下放被除数后续的3个数字；

若此时被除数仅有两个数字可被下放，比如10，那么结果直接就是110；

被除数末尾的那两个数字不用异或处理，因为结果有效位少于CRC除数，计算终止；

End of 5.1.

 

5.2. Multiple Access Links and Protocols

  • 广播链路由多个发送方和接受方共享：任何节点传输一个帧，信道将广播该帧，使得所有其它节点收到一个副本；

  • 多个节点同时传输帧时将发生碰撞，导致碰撞帧全部丢失，碰撞时间内广播信道被浪费；

  • 因此需要多路访问协议 (Multiple Access Protocol)；

Channel Partitioning Protocol

  • 信道划分协议

  1. 时分多路复用 TDM

    将时间划分为时间帧，并进一步将每个时间帧划分为 N 个时隙 (N 为节点数)；

    通常要求每个时隙能完整传输一个链路层帧；

    每个节点只在属于自己的时隙传输数据；

  2. 频分多路复用 FDM

    将信道划分为 N 个不同的频段；

    注意每个频段带宽仍然只有 R/N bps (涉及通信知识)；

  3. 码分多址 CDMA

    主要用于无线信道；特点为即使多个节点同时发送数据也不会发生碰撞；

Random Access Protocol

  • 随机接入协议

  1. 时隙 ALOHA 协议

    时间直接被划分为时隙 (每个时隙能完整传输一个链路层帧)，节点只在时隙开始时传输帧；

    发生碰撞时，节点将以概率 p 在后续每个时隙尝试重传，直到成功；

  2. ALOHA 协议

    不再划分时间，节点想传就传；

    发生碰撞后，节点将以概率 p 在传输完当前帧后尝试重传，直到成功；

  3. 载波侦听多路访问 (Carrier Sense Multiple Access, CSMA)

    节点只有在侦听到信道一段时间没有传输时，才传输自己的数据；

    仍可能发生碰撞 → 取决于端到端信道传播时延；

  4. 具有碰撞检测的载波侦听多路访问 CSMA/CD (Collision Detection)

    基于CSMA；

    当节点侦听到信道空闲时便开始传输，如果传输时检测到了碰撞则立即停止传输 (与ALOHA的区别)；

    中止传输后，节点等待一个随机时间量，然后继续侦听信道；

    → 如何选择这个随机时间量？以太网使用二进制指数后退算法；

Taking-Turns Protocol

  • 轮流协议

  1. 轮询协议 polling protocol

    主节点轮流通知节点可以发数据了，最多发几帧；

    主节点检测到信道空闲后 (上个节点发完了或者没发)，就通知下一个；

    • 优点：无碰撞，效率搞；

    • 缺点：中心化(主节点寄了网络就寄了)，有轮训时延 (节点想要发数据得等待主节点的通知)；

    蓝牙 (Bluetooth) 协议就是轮询协议，比如手机作为主节点轮训蓝牙耳机，手表等；

  2. 令牌传递协议 token-passing protocol

    简单来讲，节点将轮流持有一个被称为令牌 (token) 的特殊帧；

    节点只有在持有token的时候才能使用信道，使用完毕后传递token；

End of 5.2.

 

5.3. Switched Local Area Networks

Link-Layer Addressing and ARP

  1. MAC地址

    也称为 LAN 地址或物理地址，6字节；

    • 适配器 (网络接口) 具有MAC地址，所以同一个主机 / 路由器可以带有多个MAC地址；

    • 链路层交换机并不具备MAC地址；

    • 协议设计中，MAC地址是全局唯一的、扁平的 (不具备层级结构)、固定的；

    • 特殊MAC地址：全1广播地址：FF-FF-FF-FF-FF-FF；

  2. 地址解析协议 (Address Resolution Protocol, ARP)

    每台主机或者路由器都维护一张ARP表，记录IP地址与MAC地址的映射对，并带有TTL (一般为20min)；

    • 如果ARP缓存命中，则直接封装成帧发送；

    • 若未命中，则构造ARP请求报文 (不要与应用层报文混淆)，带有信息源IP、目的IP和源MAC，然后被封装到链路层帧中，目标MAC地址为全1广播地址；匹配的目的主机回发ARP响应报文；

所以在跨子网传输时，MAC地址将被不断重写；

Ethernet

  • 以太网：主流的有线局域网；

    • 无连接，发送方网络适配器封装成帧后直接发送，没有预先握手；

    • 不可靠，CRC校验不通过直接丢弃，重传是TCP的事；

  • 早期以太网：同轴电缆 + 转发器 (类似于红石中继器) + CSMA/CD协议 → 所有节点共享信道；

  • 基于交换机的以太网：MAC地址表定向转发 + 交换机输出缓存 + 全双工通信 → 星形拓扑；CSMA/CD协议几乎不用；碰撞几乎消失；

  1. 以太网帧结构

    ；

    • 数据：承载上层协议分组 (如IP数据报)；

      • 以太网MTU为1500字节 → IP数据报超过1500字节就要分片；

      • 最小为46字节 → IP数据报小于46字节就要填充 (使用IP首部中的长度字段标识)；

    • 目的地址：即目的MAC地址 (包括全1广播MAC地址)；

    • 源地址：源MAC地址；

    • 类型：指示上层协议 (比如数据字段为ARP报文时，类型字段为0806)；

    • CRC纠错码；

    • 前同步码 (8字节)：前7字节都是10101010，最后一个字节是10101011；

  2. 链路层交换机

  • 过滤和转发

    依托于交换机表 (表项为 [MAC地址 - 输出接口 - 已放置时间] )，但是与路由器转发行为不完全一致：

    • 如果帧的目的MAC地址对应的输出接口和帧到达的接口一致，交换机丢弃丢弃该帧 (过滤)；

    • 交换机表没有与帧目的MAC地址匹配的表项，则广播该帧；

    • 有对应表项且到达接口和输出接口不一致，将帧放入对应接口的输出缓存 (正常转发)；

  • 自学习

    • 交换机表初始为空；

    • 交换机会记录每个入帧的源MAC和到达接口；

    • 老化期后，没有再次收到该源地址的帧，则删除这个表项；

  • 全双工：交换机的接口能同时接受和发送；

End of 5.3.

 

5.4. Virtual Local Area Networks (VLANs)

  • 经交换机组织的交换局域网具有3个缺点：

    1. 缺乏流量隔离；

    2. 硬件资源的浪费；

    3. 用户管理困难；

  • 故一般采用支持虚拟局域网 (VLAN) 的交换机；

Port Based VLAN

  • VLAN允许网络管理员在单一的物理交换机上定义多个逻辑上的虚拟局域网；交换机内部维护一张“端口到VLAN”的映射表；

    • 端口 2~8 划分给电气工程系；端口 9~15 划分给计算机科学系；端口 1 和 16 暂未分配；

    • 在同一个 VLAN 内的主机可以正常通信，而不同 VLAN 之间在二层 (数据链路层) 是完全隔离的，仿佛连接在不同的独立交换机上一样；

    • 每个 VLAN 形成一个独立的广播域；即一个端口产生的广播流量，只能到达同一个 VLAN 内的其他端口；

  • 不同VLAN如何通信 → 路由器转发；不过现代做法是将 VLAN 交换机和路由器功能集成在单一设备中；

VLAN Trunking

  • 使用多台支持VLAN的交换机定义多个虚拟局域网；(原因：物理距离限制等等)

多交换机互联瓶颈：

  • 假设两台交换机，都有EE VLAN和CS VLAN，只能在每台交换机上为每个 VLAN 单独拿出一个端口，用一根网线一一对应地连接起来；

  • 但是这种做法不具备好的可拓展性，试想交换机和VLAN数量都很多，每台交换机上的VLAN数量各不相同，最终布线将极其繁琐；

  • 干线连接：

    • 每台交换机都配置一个特殊的Trunk端口，这个端口属于这个交换机上的所有VLAN；

    • 新问题：交换机如何得知来自Trunk端口的分组属于哪个VLAN？

      → 使用VLAN tag

      出Trunk口时，交换机插入4 字节的 VLAN 标签 (VID 占 12 bit，最多支持 4096 个 VLAN)；

      交换机接受到来自Trunk端口的帧后，解析并剥离tag，再发往对应的VLAN；

    比如广播帧只在对应VLAN中广播，不区分VLAN的话，也就没法做到定向广播；

MAC Based VLAN

  • 在上述基于端口的VLAN中，交换机只看流量来自于哪个端口和该端口所在的VLAN；

  • 但在基于MAC的VLAN中，设备网卡MAC地址和对应的VLAN绑定，交换机维护了MAC和VLAN的映射表；

    • 好处是设备怎么更换物理位置都不会改变其所在的VLAN；

    • 坏处是需要人工配置新设备的MAC，移动设备的随机MAC技术也有悖于该设计；

End of 5.4.
